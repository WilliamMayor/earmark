# Budget Tool MVP - Development Plan

## Executive Summary

This document outlines the development plan for the Minimum Viable Product (MVP) of Budget Tool, a self-hosted envelope budgeting application. The MVP focuses on core functionality to validate the product concept and provide immediate value to users while establishing the technical foundation for future enhancements.

## MVP Scope Definition

### Included Features

1. **CSV Import** - Basic bank transaction import functionality
2. **Envelope Management** - Create, and edit budget envelopes
3. **Transaction Allocation** - Single transaction to envelope allocation interface
4. **Fund Transfers** - Move money between envelopes
5. **Month View** - Month-by-month budget visualization

### Explicitly Excluded from MVP

- Envelope goals and targets
- Automated funding or allocation rules
- Transaction splitting functionality
- Advanced reporting and analytics
- Bank format configuration
- Authentication/authorization
- Data export features

# Pydantic models
class Envelope(BaseModel):
    id: str
    name: str
    notes: str
    balance: Decimal

class Transaction(BaseModel):
    id: str
    date: date
    description: str

class Transfer(BaseModel):

### Phase 3: Envelope Management (Week 3)

**API Endpoints**:
- `GET /api/months/{year}/{month}` - Get month summary with all envelopes and balances
- `GET /api/months/{year}/{month}/envelopes` - List all envelopes with balances for the month
- `GET /api/months/{year}/{month}/envelopes/{id}` - Get specific envelope details for month
- `POST /api/envelopes` - Create new envelope (appears in all months)
- `PUT /api/envelopes/{id}` - Update envelope details
- `GET /api/months/{year}/{month}/envelopes/{id}/transactions` - Get envelope transaction history for month
- `POST /api/months/{year}/{month}/envelopes/{id}/transactions` - Create a new transaction/transfer for an envelope in month

**Business Logic**:
- Default "Cash" and "Pending" envelope creation on first run
- Envelope name validation (unique, alphanumeric)
- Balance calculation from ledger entries up to end of specified month
- Calculate opening balance for month (closing balance of previous month)
- Prevent deletion of non-empty envelopes
- Pending" envelope holds unallocated transactions from CSV imports
- Month navigation (current month by default)

### Phase 6: Fund Transfers (Week 6)

#### 6.1 Transfer Backend (TDD Approach)

**API Endpoints**:
- `POST /api/months/{year}/{month}/transfers` - Create transfer between envelopes for specific month
- `GET /api/months/{year}/{month}/transfers` - List transfer history for month
- `DELETE /api/transfers/{id}` - Delete/reverse a transfer

**Ledger Recording**:
Transfers are recorded as double-entry transactions in the ledger file, showing the movement of funds from one envelope to another. Each transfer creates a transaction entry with:
- Debit to the destination envelope (positive amount)
- Credit from the source envelope (negative amount)
- Description indicating it's a transfer
- Timestamp and any user-provided notes

**Validation**:
- Valid envelope IDs
- Positive transfer amounts
- Atomic ledger updates
- Allow negative envelope balances (overdrafts permitted)

## Risk Mitigation

### Technical Risks
1. **Ledger file corruption**
   - Mitigation: Atomic writes, backup before operations
   
2. **CSV parsing failures**
   - Mitigation: Preview step, error handling, skip invalid rows
   
3. **Performance with large files**
   - Mitigation: Streaming processing, pagination

### Schedule Risks
1. **Ledger format complexity**
   - Mitigation: Start simple, iterate on format
   
2. **Frontend/backend integration**
   - Mitigation: Define API contract early, use TypeScript
