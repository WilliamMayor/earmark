# Phase 2: Ledger Integration & Data Models

## Overview
This phase establishes the core data layer of the Budget Tool by implementing plain text accounting using the ledger/hledger format. It involves creating the data models, implementing ledger file operations, and establishing the foundation for all financial data management.

## Duration
1 week (Week 2)

## Objectives
- Implement ledger file reading and writing operations
- Create Pydantic data models for all core entities
- Design and implement the ledger file format
- Establish double-entry bookkeeping patterns
- Create the foundation for transaction and envelope management
- Implement comprehensive testing for data integrity

## Deliverables

### 1. Data Models

#### Core Models (Pydantic)
```python
class Envelope(BaseModel):
    id: str
    name: str
    notes: str
    balance: Decimal
    created_at: datetime

class Transaction(BaseModel):
    id: str
    date: date
    description: str
    amount: Decimal
    envelope_id: str  # Always assigned to an envelope

class Transfer(BaseModel):
    from_envelope: str
    to_envelope: str
    amount: Decimal
    date: date
    description: Optional[str]
```

#### Supporting Models
- TransactionLine (for ledger entry lines)
- LedgerEntry (complete ledger transaction)
- AccountBalance (account state)
- ParseError (for error handling)

### 3. Ledger File Format

#### Structure
```
; Budget Tool Ledger File
; Generated by Budget Tool MVP
; Version: 1.0

; === Accounts Declaration ===
account Envelopes:Cash
account Envelopes:Pending
account Assets:Bank
account Equity:Opening

; === Opening Balances ===
2024-01-01 * Opening Balances
    Envelopes:Cash                 £0.00
    Envelopes:Pending              £0.00
    Equity:Opening

; === Transactions ===
; Bank import from CSV
2024-01-15 * "Grocery Store" ; imported: 2024-01-20
    Envelopes:Groceries           £45.23
    Assets:Bank                  -£45.23

; Transfer between envelopes
2024-01-20 * "Transfer: Cash to Groceries"
    Envelopes:Groceries           £100.00
    Envelopes:Cash               -£100.00

; Manual transaction entry
2024-01-25 * "Cash withdrawal"
    Envelopes:Cash                £50.00
    Assets:Bank                  -£50.00
```

#### Format Rules
- UTF-8 encoding
- ISO date format (YYYY-MM-DD)
- Decimal amounts with 2 decimal places
- Account names use colon hierarchy
- Comments preserve metadata
- Cleared transactions marked with *

## Risks and Mitigations

### Risk 1: Ledger Format Complexity
**Risk**: Ledger format has many edge cases and variations
**Mitigation**: Start with minimal subset, validate against hledger, incremental feature addition

### Risk 2: File Corruption
**Risk**: Concurrent access or crashes could corrupt ledger file
**Mitigation**: Atomic writes, automatic backups, file locking, recovery procedures

### Risk 3: Performance with Large Files
**Risk**: Parsing large ledger files could be slow
**Mitigation**: Implement caching, lazy parsing, consider indexing for future

### Risk 4: Floating Point Precision
**Risk**: Financial calculations losing precision
**Mitigation**: Use Decimal type throughout, enforce 2 decimal places, comprehensive testing
